# [Guide] 운영일지 플러그인

## 🔹1. 플러그인 목적: 문제 정의 (페인 & 솔루션)

- **페인(Pain):** 파편화된 교육 운영 데이터(세일즈맵·캘린더·슬랙·Gmail·드라이브)를 매번 직접 추적하다 놓칠까봐 불안한 것. 즉 "오늘 챙겨야 할 일이 빠진 건 없나"를 LD가 머릿속·여러 시트·메일을 일일이 대조해야 하는 부담이 큼.
- **솔루션(Solution):** 파편화된 교육 운영 데이터를 한 판에 모아 놓쳐서는 안 될 것을 눈에 띄게 보여 주는 도구. /운영일지 한 번 실행하면 오늘 챙겨야 할 영역과 누락된 부분이 보이는 MD 리포트 생성.

**사고방식:** 해석·집약·강조 ❌ → 누락(빈칸) 감지 ✅

## 🔹2. 작동 원리 개요

### 5개 데이터 소스 통합:

- **세일즈맵 본부장 DB (메인/출발점) —** 매일 새벽 3시 KST 자동 갱신. 진행 중인 Won 교육만 골라(edu_end>=오늘) 운영 대상 결정. 교육 시작·종료일, 고객사명, 차수 정보, 교육 형태(출강·복합) 등 결정론적 데이터 회수.
- **구글 캘린더 (메인/일정 보강) —** LD 본인 캘린더 + 팀 공용 캘린더 (settings에 ID 박음·옵션). 세일즈맵 일정이 비어있거나 다차수 회차별 일정이 필요할 때 보강. [고객사] 교육명 패턴 매칭**.**
- **슬랙 (증거 메인/raw) —** 운영 채널 P1/P2/P3 우선순위 구조. P1(팀 운영 요청·논의 채널)이 1차 증거 소스, P2(강사·매칭 채널)는 강사·매칭 증거, P3(팀 전체 채널)은 보충. 운영 요청 BOT thread 자동 수집으로 다차수 회차 분해도 회수 가능하나 이 부분에서 오류 발생 여지 있음.
- **Gmail (증거 보조) —** LD 본인 메일함. 체크포인트별 증거 키워드(계약·교안·세금계산서·만족도 등)로 메일 제목·본문 메타 수집.
- **구글 드라이브 (증거 보조) —** 기획·싱크업·교안·계약서 폴더에서 파일명 키워드 매칭. 사전 확인서(고객사 교육 환경)·강사 사전 확인서·교안 PPT 등 증거 회수.

### 파이프라인 흐름:

세일즈맵에서 진행 중인 Won 딜 회수 → 일정 조립(5개 데이터 소스 병합) → 교육-세션 × 9 체크포인트 격자 생성 → 오늘 도래 셀만 추출 → 셀별 증거 수집 → LLM 5분류 라벨 판정 → 3구역 리포트 출력. LLM은 ‘증거 정리자’ 역할만 — 자유 생성·해석·서술 없이 ✅/🟡/🔴/⚪ 라벨 하나만 출력합니다.

## 🔹4. 플러그인 설치

플러그인 코드와 본인 환경 설정 파일을 받아오는 단계입니다. 1)·2)는 별도 터미널에서, 3)은 Claude Code 대화창에서 진행합니다.

1. **플러그인 코드 받기**

```bash
git clone https://github.com/D1-B2B-AX/ld-ops-plugin ~/.claude/commands/ld-ops-plugin
```

- ~/.claude/commands/ld-ops-plugin 폴더가 생기면 성공
- 이후 /운영일지 실행할 때마다 자동으로 최신 버전을 받아옵니다 (직접 git pull 불필요)

```bash
cp ~/.claude/commands/ld-ops-plugin/config/settings.example.json ~/.claude/commands/ld-ops-plugin/config/settings.json
```

- settings.json은 개인정보(이메일·슬랙 user ID·캘린더 ID 등)가 들어가서 GitHub에 안 올라가게 차단됩니다. 그래서 본인이 직접 만들어야 합니다.

## 🔹5. 9개 운영 체크포인트는 어떻게 도출됐나 (설계 흐름)

본 plugin의 9개 체크포인트는 세 영역에서 도출됐습니다 — 운영 표준 WBS 필터링 + LD 인터뷰 컴펌 내용 반영.

### 1) 프레임 설정

운영의 본질은 해석·집약·강조가 아니라 **누락(빈칸) 감지**라는 점을 중점으로, LLM 역할을 ‘판단자’에서 ‘증거 정리자’로 축소. 자유 생성·서술 제거하고 체크포인트 단위 라벨 판정만 담당하도록 구조 재설계.
**즉, 누락된 내용의 핵심만 보여주지 모든 운영 데이터를 보여주진 않습니다!**

### 2) 운영 표준 WBS 필터링

조직 공식 운영 템플릿(WBS) 50개 업무 중 LD 관점에서 필수로 처리해야 하는 항목만 추렸습니다 (초안 7개). 

### 3) LD (강연정) 인터뷰 컴펌 내용 반영

**WBS 기반 초안을 교육 2팀 LD 강연정에게 검증받아 필수 7 + 조건부 2 = 9개 구조로 확정.** 

- D-N 표기 — 리스크 알림=발화 기준 / 과정별 알림=단순 표시 (이중 구조)
- 기한 관리 — 교육일 기준 역산 + 특이 건(슬랙·싱크업 시트) 우선
- 알림 시점 — 사전 경고 (당일은 리스크 큼)

세 영역이 결합되어 현재 plugin의 **필수 7 + 조건부 2 구조 + 시점 역산 윈도우(window) + severity 3단계(즉각해결·확인필요·관찰)**로 박혀 있습니다.

## 🔹6. 셀 매트릭스 + 5분류 라벨 — 오늘 봐야 할 셀만 골라내는 구조

**“교육마다 할 일 목록(9개 체크포인트)을 두고, 오늘(각 체크포인트 별 확인해야 하는 일반적인 체크일 선에서) 진짜 체크해야 할 항목만 골라 본다”는 구조**입니다.

### ▪️ 셀 매트릭스 구성

교육마다 9개 체크포인트로 체크리스트를 만듭니다. 다차수 교육은 회차별로 별도 체크리스트가 박힙니다. 단 체크포인트마다 단위가 다른데, 딜 전체에 1번만 체크하는 항목(기업계약·세금계산서 등)과 회차마다 따로 체크하는 항목(교안 컨펌·교육생 안내 등)으로 구분됩니다.

### ▪️ 도래 판정 — window로 오늘 셀 고르기

**각 체크포인트마다 “언제쯤 돼 있어야 한다”는 시점 기준(window: from D-X \~ to D-Y)이 정해져 있습니다.** 오늘 날짜를 이 window에 대보면 셀마다 세 상태가 나와요:

- 아직 시점이 안 온 셀 → 📅 시점 미도래 (오늘 볼 필요 없음)
- 시점에 들어온 셀 → 증거 확인 대상
- 세션도 끝났고 시점도 오래 지난 셀 → 이제 와서 봐도 고칠 수 없는 과거 → 동일하게 볼 필요 없음 (Stale 필터 자동 제외)

### ▪️ LLM 역할 — 도래 셀 × 증거 있음 에만

window 기준 별 확인 대상으로 걸러진 셀은 해당 체크포인트와 관련된 슬랙·메일·드라이브 증거를 찾는데, 증거가 전혀 없으면 LLM 호출도 없이 자동으로 🔴 처리. 증거가 있을 때만 LLM이 ✅/🟡/🔴/⚪ 중 하나를 붙입니다. **즉 LLM은 오늘 볼 셀 중에서도 증거가 실제로 있는 셀에만 작동.**

### ▪️ 5분류 라벨

- **📅 시점 미도래** — 아직 체크해야 할 때가 아닌 셀 (표시만, 도래 X)
- **✅ 완료 추정** — 완료 증거가 확인된 셀
- **🟡 진행 중** — 논의 시그널은 있으나 완료 증거 부재
- **🔴 미확보** — 증거 자체가 없어 즉각 확인 필요
- **⚪ 모호** — 증거는 있으나 완료/미완료 판정 애매 (사람 확인 필요)

<aside>
💡 **핵심 이점 — 이렇게 바뀌면서 전에는 모든 딜을 통째로 LLM에 넘겨 해석시키던 방식이 ‘필요한 셀만, 필요한 일만’ 구조로 바뀌어, 토큰·속도·일관성 문제가 동시에 완화됩니다.**

</aside>

## 🔹7. 9개 체크포인트 상세 — 각 체크포인트가 어떤 운영 본질을 잡나

**필수 7개는 모든 교육 공통·조건부 2개는 딜별 플래그로 on/off**. 알림 단위(granularity)는 딜 단위(다차수 교육이라도 딜당 1번만 알림)인지 세션 단위(회차마다 별도 알림)인지 구분합니다.

### 🔸1번. 기업계약 체결 (필수, 딜 단위)

고객사 계약 서명 완료 증거(계약서 파일 또는 서명 완료 메시지·메일). 

☑️시점: 교육 확정 즉시 ~ D-7. ☑️severity: 확인 필요.

### 🔸2번. 강사계약 체결 (필수, 딜 단위)

강사 계약 서명 완료 증거. 섭외는 제안 단계에서 이미 완료된다는 전제. 

☑️시점: D-7 쯤 (기업 계약 체결 후). ☑️severity: 즉각해결.

### 🔸3번. 싱크업 미팅 — LD→OM 인계·다차수 (필수, 딜 단위)

LD와 OM 간 싱크업 미팅 진행 증거 (싱크업 문서 미팅로그 탭 기재 또는 슬랙 싱크업 논의 스레드). 다차수 교육은 회차별 정기 추가.

 ☑️시점: LD→OM 인계 즉시 (D-30 이전). ☑️severity: 확인 필요.

### 🔸4번. 고객사 교육 환경 확인 — 강의장·장비·보안 (필수, 딜 단위)

고객사 강의장·장비·보안 사전 확인 회신 증거 (기업용 사전 확인 요청서 회신 또는 고객사 확인 메일).

 ☑️시점: D-21 ~ D-14 (최소 D-10). ☑️severity: 즉각해결.

### 🔸5번. 강사 교안·교육 환경 요청/검수 (필수, 딜 단위)

강사에게 교안·교육 환경(설치 프로그램·보안 접속 등) 요청·회신 증거 (강사 사전 확인 요청서 회신).

 ☑️시점: D-21 쯤. ☑️severity: 확인 필요.

### 🔸6번. 교안 컨펌 + 고객사 전달 (필수, 세션 단위)

강사 교안 컨펌 완료 + 고객사에 교안·접속 사이트·설치 프로그램 전달 증거. 

 ☑️시점: D-10 ~ D-7. ☑️severity: 즉각해결.

### 🔸7번. 거래명세서·세금계산서 발행 (필수, 딜 단위)

거래명세서 또는 세금계산서 발행 증거 (계산서 발급 신청·발행 완료 메시지·거래명세서 송부 등). 다차수라도 딜 1회만 발행. 어느 한쪽이라도 처리되면 진행 단서로 인정.

 ☑️시점: D+3 ~ D+7 (거래명세서 선행). ☑️severity: 관찰.

### 🔸8번. 교육생 입과·강의 안내 메일 (조건부, 세션 단위)

학습자에게 교육 입과/강의 안내 메일 발송 증거. 보통은 고객사가 담당하는 영역이라 기본은 평가 안 함. 우리 쪽이 담당하는 딜만 켜서 평가 (Claude Code에 *"이 딜은 우리가 담당이야"*라고 시키면 자동으로 settings에 박힘)

 ☑️시점: D-7. ☑️severity: 확인 필요.

### 🔸9번. 만족도 리포팅 — 기업·강사 (조건부, 세션 단위)

만족도 설문 결과를 기업·강사에게 리포팅한 증거 (리포팅 메일 발송 또는 결과 파일 공유). 보통은 우리 쪽이 담당하는 영역이라 기본은 평가함. 만족도 리포팅 안 하는 딜만 꺼서 평가 제외 (Claude Code에 *"이 딜은 만족도 리포팅 안 해"*라고 시키면 자동으로 settings에 박힘)

 ☑️시점: D+3 ~ D+7 (늦어도 D+14). ☑️severity: 확인 필요.

## 🔹9. 커스터마이징

플러그인은 특정 LD 환경에 맞춰 박혀 있지만, **모두 본인 환경으로 수정 가능**합니다. 
핵심은 settings.json — 다른 LD는 본인의 채널·라벨·캘린더로 바꾸면 그대로 작동합니다. 모든 수정은 Claude Code 대화창에 자연어로 시키면 즉시 반영됩니다.

### (1) settings.json — 본인 환경 수정 (메인)

- owner(이름·회사 이메일·팀·슬랙 user ID·캘린더 ID)
- gmail_label — 본인 메일 라벨 (deploy 모드 시엔 본인 메일함 자동)
- slack_ops_team_channels — 팀별 운영 채널 (P1·P2·P3 구조)
- ops_included_course_formats — 운영 체크포인트 적용 교육 형태 (기본: 출강·복합)

자연어 예시: "내 슬랙 채널에 #b2b_3팀_운영 추가해줘" / "내 회사 이메일·이름 박아줘"

### (2) 체크포인트 수정 — checkpoints.json

9개 체크포인트의 시점 window·severity·증거 키워드 등은 config/checkpoints.json에 박혀있습니다. 자연어로 조정 가능:

- 시점 조정: "교안 컨펌 시점을 D-7에서 D-10으로 올려줘"
- severity 조정: "기업계약을 즉각해결로 올려줘"
- 증거 키워드 추가: "교안 체크포인트 키워드에 ‘검토안’ 추가해줘"

### (3) 딜별 플래그 조정 (조건부 체크포인트 on/off)

- "이 딜은 교육생 안내 메일 우리가 담당이야"
- "이 딜은 만족도 리포팅 안 해"

### (4) 본인 깃 fork·공유 (선택)

본인 수정 영역을 보존하거나 다른 LD와 공유 원할 때만:

- 본인 깃에 보존: "내 GitHub에 fork해서 올려줘" → Claude Code가 본인 fork에 push
- 다른 LD와 공유: 본인 fork URL 슬랙 공유
- AX팀 main 반영: PR 보내거나 슬랙으로 요청

## 🔹10. 추후 보완 가능 영역 (v2)

본 plugin v1은 체크포인트 기준 매트릭스 뼈대를 박은 영역입니다. 다만 **설계 과정에서 운영 관리 방식이 LD마다 편차가 크다는 본질적 한계가 드러났습니다** — 같은 운영이어도 어떤 데이터를 메인으로 삼는지·어디까지 시스템에 박는지가 LD마다 다르고, 본 plugin이 표준으로 박은 체크포인트·시점 기준(window)도 사실은 **개인 운영 스타일·딜별 상황에 따라 조정 여지가 큰 영역**입니다. 이런 부분은 LD/딜별 customization 보강이 필요한데, 추후 검토 영역으로 두고 v1엔 표준 뼈대 중심으로 박았습니다.
