# 🎼 ops-plugin — 오케스트레이터 (운영 전용)

> LD가 **교육 운영 체크포인트의 빈칸(준비 증거 없음)을 기한 전에 찾아내도록**, 공통 layer(core) + 체크포인트 매트릭스 + 증거 수집·LLM 분류를 지휘하여 운영일지 MD를 생성하는 메인 스킬.

## 버전

- **v1.0 (2026-04-24)**: 260424 전환 완료. "해석형 → 누락 감지형" 구조 전면 재설계. 이전 버전은 `archive/260424_pivot/` 보관.
- v0.5 (2026-04-23): LLM 자유 생성 제어 시도 — 폐기
- v0.1 (2026-04-22): 뼈대 수립 — 폐기

## 🎯 원칙 0 — 북극성

**페인:** "파편화된 교육 운영 데이터를 매번 직접 추적하다 놓칠까봐 불안한 것"

**솔루션:** "준비 완료돼 있어야 할 시점에 아직 증거가 없는 것을 기한 전에 알려주는 도구"

이 도구는 **누락 감지형**. 해석·판단·생성 X. **빈칸 드러내기 O.**

## 핵심 원칙 6가지 (전환 후)

1. **LLM은 증거 정리자** — 자유 생성 금지. 증거 4분류 라벨(✅/🟡/🔴/⚪)만 판정
2. **결정론 우선** — 일정 조립·격자 만들기·리포트 배치는 전부 스크립트 결정론
3. **체크포인트 매트릭스** — 교육-세션 × 체크포인트 9개 격자. 오늘 도래한 셀만 처리
4. **앞단 조립 레이어** — Step 0.5 일정 조립으로 단일 날짜 모델 한계 극복
5. **원칙 0 우선** — 모든 설계 결정은 페인·솔루션 부합 여부로 검증
6. **MVP 최소주의** — 불필요한 건 v0에서 제거, 필요성 확인되면 v1 추가 (예: OM 매칭·투두 자동생성은 v0 제외)

## 실행 조건

사용자가 `/운영일지` 실행 시 본 오케스트레이터가 순서대로 수행.

**권장:** `claude --dangerously-skip-permissions`로 시작하면 중간 승인 없이 자동 실행.

## 전체 흐름 (6단계)

```
Step 0. 재료 준비 [기존 유지, data-skills/1~5]
  Won 딜 수집 + 5소스 데이터 (세일즈맵·캘린더·슬랙·지메일·드라이브)
       ↓
Step 0.5. 일정 조립 [compose_schedule.py, 신설]
  4소스 병합으로 교육-세션별 sessions[] 조립
  세일즈맵·캘린더(결정론) + 드라이브·슬랙(좁은 LLM 날짜 추출)
       ↓
Step 1. 셀 매트릭스 구축 [build_matrix.py, 신설]
  교육-세션 × 체크포인트 9개 격자
  anchor + window + 오늘 날짜로 "도래 셀" 추출
  deal_flags 확인 (조건부 체크포인트 활성화 판정)
       ↓
Step 2. 증거 수집 [collect_evidence.py, 신설]
  도래 셀마다 evidence_keywords로 슬랙·메일·드라이브 검색
  체크포인트 특화 키워드 + 기존 필터(Owner Name 멘션·라벨·폴더) 결합
       ↓
Step 3. LLM 분류 [classify_evidence.py, 신설]
  증거 + completion_hint → ✅/🟡/🔴/⚪ 라벨 1개
  자유 생성 금지. JSON 응답 구조 고정
       ↓
Step 4. 리포트 생성 [generate_ops_md.py, 재설계]
  severity 기준 배치:
    🔴 즉각해결 → 상단 리스크 섹션
    🟡 확인필요 / ⚪ 관찰 → 중단 교육-세션별 행 뷰
    ✅ 완료 → 하단 로그 (또는 숨김)
       ↓
Step 5. 자연어 피드백 루프 [apply_feedback.py, 유지]
  LD 입력 → 4종 조작 (셀 상태 override / 일정 이동 / watchlist 제외 / OM 수정)
  state/ops_state.json JSONL 축적, 다음 회차 반영

출력: outputs/ops_report_YYYYMMDD.md + .slack.txt
```

## Step별 상세

### Step 0 — 재료 준비 (5소스, 2팀은 노션 포함)

`data-skills/1~5` **병렬 호출** + 2팀 한정 스킬 6 노션. 스킬별 SKILL.md 그대로.

- 스킬 1: 세일즈맵 Won 딜 조회 (`status='Won' AND edu_end >= today`)
- 스킬 2: 캘린더 이벤트 (`team_education_schedule` 단일 캘린더, `[고객사]` 브래킷 패턴)
- 스킬 3: 슬랙 (운영 채널, owner `from:`/멘션 필터 강제)
- 스킬 4: 지메일 (test/deploy 2모드)
- 스킬 5: 드라이브 기획·싱크업
- **스킬 6 노션 강의 캘린더 (2팀 전용)** — `owner.team`이 "교육 2팀" 또는 "교육 2팀 운영 파트"일 때만. v1.0 MVP는 CSV 수동 배치 + `parse_notion_csv.py` 파싱 (`settings.data_sources.notion_schedule_csv_path_by_team` 참조)

**⚡ 병렬화 필수:** 스킬 2·3·4·5는 독립이므로 **한 번의 function_calls 블록 내 4 tool 호출 동시 투입.**

**⚡ 스킬 1 내부 메모 파싱도 반드시 병렬:** 스킬 1 STEP 3(LLM 메모 파싱)은 딜별 LLM 호출을 **한 번의 function_calls 블록에 N딜 동시 투입** (`data-skills/1-salesmap-조회/SKILL.md` §STEP 3 v0.5 병렬화 규칙 참조). 4딜 10초 초과 시 순차 의심 → 시간 재측정·로그 확인. 각 딜 메모는 독립 텍스트라 병렬 안전.

**결과 병합 형식 (중요):**
각 스킬은 **딜 1건당 1개 객체**를 반환. 여러 딜 병렬 호출 후 오케스트레이터가 결과를 `{deal_id: result}` 딕셔너리로 묶어 runtime/ 에 저장:

- `runtime/s1_deals.json` — `{"deals": [{deal_id, ...}]}` (스킬 1이 이미 배열로 반환)
- `runtime/s2_calendar.json` — `{"<deal_id>": {deal_id, deal_name, matched_events: [...]}}`
- `runtime/s3_slack.json` — `{"<deal_id>": {deal_id, deal_name, slack_results: [...], slack_summary, ...}}`
- `runtime/s4_gmail.json` — `{"<deal_id>": {deal_id, deal_name, thread_summary: [...], email_parsed, ...}}`
- `runtime/s5_drive.json` — `{"<deal_id>": {planning_sheet: {...}}}`

Python 스크립트들은 이 딕셔너리 구조를 기대. 병합 형식 틀리면 Step 0.5부터 실패.

### Step 0.5 — 일정 조립 (신설, v1.0)

**목적:** 단일 날짜(edu_start) 모델 한계 극복. 다차수·일정 변경·세일즈맵 공백 케이스 대응.

**처리 순서:**

1. **유형 1차 판정** (세일즈맵 기간 기반)
   | 기간 | 유형 | 처리 |
   |---|---|---|
   | ≤7일 | `single` 단차수 | 세일즈맵 단일 세션 + 캘린더/드라이브 보강 |
   | 7~14일 | `ambiguous` 모호 | 캘린더 이벤트 ≥2 → 다차수, 아니면 단차수 |
   | >14일 | `multi` 다차수 | 캘린더 이벤트를 회차로, 부족 시 드라이브 분해 |
   | 공백 | `unknown` 미상 | 드라이브 → 캘린더 순 복원 |

2. **4소스 병합** (유형별 규칙)
   - 세일즈맵 · 캘린더 = 결정론 (DB/API 필드)
   - 드라이브 `session_schedule[]` = 좁은 LLM (탭 파싱, Step 0에서 수행됨)
   - 슬랙 공지 = 좁은 LLM (**PENDING으로 분리, 오케스트레이터 후속 처리**)

3. **충돌 해결** — 기본은 세일즈맵·캘린더 공식. 단 특이 건(일정 변경 감지 시)은 슬랙/드라이브 우선

**슬랙 공지 처리 (중요 — 오케스트레이터 역할):**

`compose_schedule.py`가 슬랙에서 일정 관련 키워드+날짜 패턴 포함 메시지를 필터링 후 **LLM 프롬프트만 준비**하고 `pending_slack_updates` 배열에 담아 반환.

오케스트레이터 후속 처리 워크플로:

```
1. sessions.json 읽고 pending_slack_updates 확인
2. status=="PENDING" 각 항목에 대해:
   - llm_prompt 필드 값을 LLM에 그대로 전달
   - JSON 응답 수집: {"sessions": [...], "no_schedule_info": bool}
3. 응답들을 하나의 JSON 파일로 저장:
   runtime/slack_responses.json
   {
     "<deal_id_1>": {"sessions": [{"edu_start": "...", "edu_end": "...", "note": "..."}],
                     "no_schedule_info": false},
     "<deal_id_2>": {"sessions": [], "no_schedule_info": true}
   }
4. 병합 실행:
   python scripts/apply_llm_responses.py --mode slack \
     --target runtime/sessions.json \
     --responses runtime/slack_responses.json
5. 결과 확인 — pending_slack_count 0 이어야 다음 Step 진행
```

**확정도 판정:**

| confidence | 조건 |
|---|---|
| `high` | 세션 평균 2+ 소스 일치 |
| `medium` | 1 소스 |
| `low` | 낮음 |
| `none` | 세션 0건 → `deals_no_schedule` 분리 |

**출력:** `runtime/sessions.json`
```json
{
  "deals_with_schedule": {
    "<deal_id>": {
      "deal_name": "...",
      "customer": "...",
      "course_id": "...",
      "session_type": "single|multi|ambiguous|unknown",
      "confidence": "high|medium|low",
      "sessions": [
        {"session_no": 1, "edu_start": "2026-04-23", "edu_end": "2026-04-23", "source": ["salesmap", "calendar"]}
      ],
      "warnings": ["세일즈맵 ↔ 캘린더 불일치 등 경고"]
    }
  },
  "deals_no_schedule": [
    {"deal_id": "...", "reason": "4소스 공백"}
  ],
  "pending_slack_updates": [
    {"deal_id": "...", "candidates": [...], "llm_prompt": "...", "status": "PENDING"}
  ]
}
```

### Step 1 — 셀 매트릭스 구축 (신설)

**입력:** `sessions.json` + `checkpoints.json` + `settings.json`의 `deal_flags`

**로직 의사코드:**
```
for deal in sessions:
  for session in deal.sessions:
    for checkpoint in checkpoints:
      # 조건부 체크포인트 플래그 확인
      if checkpoint.flag_id and not deal_flags[checkpoint.flag_id]:
        continue
      
      # 날짜 계산
      anchor_date = session[checkpoint.anchor]  # edu_start or edu_end
      today_offset = (today - anchor_date).days
      
      # window 내 판정
      if checkpoint.window.from <= today_offset <= checkpoint.window.to:
        matrix.append({deal, session, checkpoint, offset, status: "arriving"})
      elif today_offset > checkpoint.window.to:
        matrix.append({deal, session, checkpoint, offset, status: "overdue"})
```

**출력:** `runtime/arriving_cells.json` — 오늘 감시 대상 셀 목록

### Step 2 — 증거 수집 (신설)

**입력:** `arriving_cells.json` + Step 0 수집 데이터 (`runtime/s3_slack.json`, `s4_gmail.json`, `s5_drive.json`)

**⚠️ MVP 구현 방식 명시 (중요):**
v0에서는 **Step 0에서 이미 수집된 데이터를 재사용해 키워드 필터링**함. **MCP 재호출 아님.**
- 장점: 구현 단순, 추가 MCP 호출 없음
- 한계: Step 0 수집이 "교육 전반"이라 체크포인트 특화 증거 놓칠 가능성
- v1 검토: 필요 시 "도래 셀만 MCP 재호출해 좁은 쿼리로 증거 수집" 방식으로 전환 (토큰 효율 ↑)

**로직:** 각 도래 셀에 대해:
1. `checkpoint.evidence_keywords.slack` 키워드로 Step 0 수집 슬랙 메시지 필터
2. `checkpoint.evidence_keywords.gmail`로 메일 스레드 필터
3. `checkpoint.evidence_keywords.drive`로 드라이브 파일명·탭명·내용 매칭
4. 매칭된 증거를 스니펫(snippet, 짧은 발췌)으로 정리

**Step 0 수집 시 쿼리 예시 (참고, 기존 자산):**
```
(from:Owner Name OR @Owner Name) Customer B 정기특강 in:운영채널들 after:2주전
```

**출력:** `runtime/evidence.json`

### Step 3 — LLM 분류 (신설)

**입력:** `evidence.json` + `checkpoints.json`의 `completion_hint`

**프롬프트 구조:**
```
체크포인트: {checkpoint.label}
완료 정의: {checkpoint.completion_hint}
교육 메타: 고객사={customer}, 과정명={course}, D-day={offset}
수집 증거:
  [슬랙] ...
  [메일] ...
  [드라이브] ...

위 증거가 체크포인트 완료의 근거로 충분한가?
다음 4개 중 정확히 하나만 선택하시오:
  ✅ 완료 증거 명확
  🟡 진행 중 흔적만 (완료 확인 불가)
  🔴 증거 전무
  ⚪ 흔적 있으나 완료 여부 모호

응답 형식: {"label": "<이모지 1개>", "confidence": <0.0~1.0>}
```

**자유 생성 금지.** 이모지 + confidence 외 응답은 🔴 fallback.

**오케스트레이터 후속 처리 워크플로 (Step 3 완료 후):**

```
1. classified_cells.json 읽고 label=="PENDING" 셀 선별
2. 각 PENDING 셀의 llm_prompt를 LLM에 전달 → JSON 응답 받음:
   {"label": "🔴|🟡|✅|⚪", "confidence": 0.0~1.0}
3. 응답들을 하나의 JSON 파일로 저장:
   runtime/label_responses.json
   {
     "0": {"label": "🟡", "confidence": 0.8, "reason": "진행 중 흔적 있음"},
     "5": {"label": "✅", "confidence": 0.95},
     "deal_001:curriculum_confirm:1": {"label": "🔴", "confidence": 0.9}
   }
   key = 셀 인덱스(숫자 문자열) 또는 "deal_id:checkpoint_id[:session_no]" 형태
4. 병합 실행:
   python scripts/apply_llm_responses.py --mode labels \
     --target runtime/classified_cells.json \
     --responses runtime/label_responses.json
5. 결과 확인 — pending_llm 0 이어야 Step 4 진행 (잔존 시 generate_ops_md가 🔴 fallback)
```

**출력:** `runtime/classified_cells.json`

### Step 4 — 리포트 생성 (재설계)

**입력:** `classified_cells.json` + `checkpoints.json` (label·severity·timing_desc)

**섹션 구성:**

```markdown
# 교육 운영일지 — YYYY-MM-DD (요일)

## 🔴 즉각 해결 필요
(severity=즉각해결 + label=🔴 셀만)
- [Customer B 1회차 · D-7] 교안 컨펌 — 증거 전무
- ...

## 📋 확인 필요 / 관찰
(severity=확인필요·관찰 + label=🟡·⚪ 셀, 교육-세션별 행)
교육-세션 | D-day | 체크포인트 | 상태 | 비고
Customer B 1회차 | D-5 | 기업 환경 확인 | 🟡 진행 중 | 4/20 슬랙 논의

## ✅ 최근 처리 (지난 7일)
(label=✅ 셀)
- 2026-04-20 [Customer B] 강사계약 체결 완료

## ⚠️ 일정 미확정
(Step 0.5에서 🔴 미확정으로 분류된 딜)
- Customer C Bootcamp Z — 4소스 다 공백

## 🎙️ 자연어 피드백
"Customer B 교안 검토 완료됐어" 같은 자연어로 셀 상태 수정 가능
```

출력 포맷 상세는 `docs/output_format_spec_v1.md` 참조 (신설 예정).

**출력:** `outputs/ops_report_YYYYMMDD.md` + `.slack.txt`

### Step 5 — 자연어 피드백 루프 (유지, 조작 재정의)

`apply_feedback.py` 인프라 유지. 피드백 대상만 §6.8 조작으로 재정의.

**v0 MVP 조작 4종:**

| 조작 | 예시 | 반영 위치 |
|---|---|---|
| 셀 상태 override | *"실제로 교안 검토 완료됨"* | `classified_cells.json` 해당 셀 label 강제 변경 |
| 일정 이동 | *"S1은 다음 주로 미뤄"* | `sessions.json` 세션 날짜 수정 |

## 전체 실행 시퀀스 (v1.0 CLI 명령)

**작업 디렉토리:** `C:\Users\GA\Desktop\ld-plugins\ops-plugin`
**참고:** Step 0(MCP 실데이터 수집)과 PENDING 처리는 **Claude Code 오케스트레이터**가 담당. 나머지 Python 스크립트는 아래 명령으로 순차 실행.

```bash
BASE="<plugin-root>"
CORE="<plugin-root>"
TODAY=$(date +%Y%m%d)

# Step 0 — 재료 준비 (Claude가 MCP 병렬 호출)
#   ↓ data-skills/1~5 병렬 호출 → runtime/s1_deals.json, s2_calendar.json,
#     s3_slack.json, s4_gmail.json, s5_drive.json 생성 (기존 자산)

# Step 0.3 — 노션 CSV 파싱 (2팀 전용, 선택)
# 사용자가 노션 내보내기 CSV를 runtime/notion_calendar.csv에 미리 배치
if [ -f "$BASE/runtime/notion_calendar.csv" ]; then
  python "$BASE/scripts/parse_notion_csv.py" \
    --csv      "$BASE/runtime/notion_calendar.csv" \
    --salesmap "$BASE/runtime/s1_deals.json" \
    --settings "$CORE/config/settings.json" \
    --out      "$BASE/runtime/s6_notion.json"
fi

# Step 0.5 — 일정 조립 (5소스 + 운영 요청 채널 thread)
NOTION_OPT=""
[ -f "$BASE/runtime/s6_notion.json" ] && NOTION_OPT="--notion $BASE/runtime/s6_notion.json"
OPS_REQ_OPT=""
[ -f "$BASE/runtime/s3_slack_ops_requests.json" ] && OPS_REQ_OPT="--slack-ops-requests $BASE/runtime/s3_slack_ops_requests.json"
python "$BASE/scripts/compose_schedule.py" \
  --salesmap "$BASE/runtime/s1_deals.json" \
  --calendar "$BASE/runtime/s2_calendar.json" \
  --drive    "$BASE/runtime/s5_drive.json" \
  --slack    "$BASE/runtime/s3_slack.json" \
  --out      "$BASE/runtime/sessions.json" \
  $NOTION_OPT $OPS_REQ_OPT
# --slack-ops-requests: Customer F형(N차 M/D~M/D)·Customer E형(M월 D, D) 다차수 분해

# Step 0.5 후속 — pending_slack_updates 있으면 Claude가 LLM 호출하여 sessions.json 병합
#   (sessions.json의 각 PENDING 항목 llm_prompt를 LLM에 전달 →
#    {"sessions": [...], "no_schedule_info": bool} 응답 → 해당 deal의 sessions 덮어쓰기)

# Step 1 — 셀 매트릭스
python "$BASE/scripts/build_matrix.py" \
  --sessions    "$BASE/runtime/sessions.json" \
  --checkpoints "$CORE/config/checkpoints.json" \
  --state       "$BASE/state/ops_state.json" \
  --out         "$BASE/runtime/arriving_cells.json"
# --today YYYY-MM-DD 옵션으로 날짜 고정 가능 (테스트용)

# Step 2 — 증거 수집 (v1.1: --settings로 슬랙 채널 P1/P2/P3 우선순위 적용)
python "$BASE/scripts/collect_evidence.py" \
  --cells       "$BASE/runtime/arriving_cells.json" \
  --checkpoints "$CORE/config/checkpoints.json" \
  --slack       "$BASE/runtime/s3_slack.json" \
  --gmail       "$BASE/runtime/s4_gmail.json" \
  --drive       "$BASE/runtime/s5_drive.json" \
  --settings    "$CORE/config/settings.json" \
  --out         "$BASE/runtime/evidence.json"

# Step 3 — LLM 분류 입력 준비 (--state로 자연어 피드백 cell_overrides 자동 적용)
python "$BASE/scripts/classify_evidence.py" \
  --evidence    "$BASE/runtime/evidence.json" \
  --checkpoints "$CORE/config/checkpoints.json" \
  --state       "$BASE/state/ops_state.json" \
  --out         "$BASE/runtime/classified_cells.json"

# Step 3 후속 (1) — PENDING 셀은 Claude가 LLM 호출하여 label 업데이트
#   (classified_cells.json의 label=="PENDING" 각 셀 llm_prompt를 LLM에 전달 →
#    {"label": "🔴"|"🟡"|"✅"|"⚪", "confidence": 0~1} 응답 → 해당 셀 label/confidence 덮어쓰기)
#   apply_llm_responses.py --mode labels 로 병합

# Step 3 후속 (2) — Claude가 서브에이전트 polish 위임 → polish_responses.json
#   (각 셀 evidence_summary를 자연 압축 brief로 — 진행 현황 데이터 표시용)
#   classified_cells.json의 polished_brief 필드에 인덱스 기반 매핑

# Step 4 — 리포트 생성 (v1.6: --checkpoints + --evidence 필수, manual_override 메시지 포함)
python "$BASE/scripts/generate_ops_md.py" \
  --classified  "$BASE/runtime/classified_cells.json" \
  --checkpoints "$CORE/config/checkpoints.json" \
  --sessions    "$BASE/runtime/sessions.json" \
  --evidence    "$BASE/runtime/evidence.json" \
  --out         "$BASE/outputs/ops_report_$TODAY.md"

# Step 4.6 — 출력 검증
python "$BASE/scripts/verify_output_format.py" \
  --md "$BASE/outputs/ops_report_$TODAY.md" --strict
# [--strict] 붙이면 검증 실패 시 exit 1 → 오케스트레이터가 Step 4 재실행 판단

# Step 4c-2 — 슬랙 호환 버전 (positional input + -o output)
python "$BASE/scripts/md_to_slack.py" \
  "$BASE/outputs/ops_report_$TODAY.md" \
  -o "$BASE/outputs/ops_report_$TODAY.slack.txt"

# Step 5 — 자연어 피드백 (선택, 리포트 검토 후)
#   Claude가 LD 자연어를 commands JSON으로 파싱 → --commands 파일 또는 stdin 입력
#   cell_override 등은 다음 실행 시 Step 3의 --state로 자동 반영됨 (영구 저장)
python "$BASE/scripts/apply_feedback.py" \
  --state    "$BASE/state/ops_state.json" \
  --commands "$BASE/runtime/feedback_commands.json"
```
